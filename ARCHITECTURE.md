## 2. LCARS Theme System

### How CSS is loaded

`landing/templates/base.html` is the **single source of truth** for all
shared assets.  It loads, in this order:

1. **Bootstrap 4** (CDN) тАУ provides the layout grid, utility classes, and
   Bootstrap-compatible modal/button HTML that crispy_bootstrap4 renders.
2. **`default.css`** тАУ custom fonts (`@font-face`), body background/colour,
   nav bar, table classes (`.sci-fi-list`), and image helpers.
3. **`lcars.css`** тАУ LCARS CSS custom properties (CSS variables) and LCARS
   component classes (panels, headings, badges, buttons).
4. **`sci-fi-form.css`** тАУ form-container and field overrides.

**Rule:** never add a `<link rel="stylesheet">` inside a `{% block content %}`
in a child template.  Put all shared styles in `base.html`.  Per-page
additional CSS goes in `{% block extra_css %}`.

### LCARS CSS variables (from `lcars.css`)

| Variable               | Purpose                         |
|------------------------|---------------------------------|
| `--lcars-bg`           | page background dark navy       |
| `--lcars-panel`        | amber panel / border colour     |
| `--lcars-panel-dark`   | darker amber for gradients      |
| `--lcars-accent`       | magenta accent (badges, chips)  |
| `--lcars-accent-2`     | cyan accent (buttons, links)    |
| `--lcars-text`         | default text colour (white)     |
| `--lcars-muted`        | semi-transparent white overlay  |

### LCARS component classes

| Class                | Usage                                               |
|----------------------|-----------------------------------------------------|
| `.lcars-panel`       | Card-like container with amber border + side nub    |
| `.lcars-heading`     | Flex row for page headings with a colour chip       |
| `.lcars-tab`         | Rounded colour block used inside `.lcars-heading`   |
| `.lcars-btn`         | Cyan gradient button                                |
| `.lcars-badge-true`  | Cyan badge for boolean Yes                          |
| `.lcars-badge-false` | Muted badge for boolean No                          |
| `.sci-fi-list`       | Dark-themed data table                              |
| `.sci-fi-form-container` | Form wrapper with LCARS panel border           |
| `.sci-fi-form-grid`  | CSS grid for multi-column form layout               |
| `.sci-fi-form-section` | One column of form fields inside the grid        |

### Template partials

| Template                       | Purpose                        |

## 3. App Conventions

### Views

All entity apps (armor, weapons, events, factions, worlds, mindmaps, party)
use **class-based generic views** (`ListView`, `DetailView`, `CreateView`,
`UpdateView`).

- `ListView` тАУ `paginate_by = 50` (or 25 for party); use
  `prefetch_related` / `select_related` in `get_queryset`.
- `CreateView` / `UpdateView` тАУ use `fields = '__all__'` or a dedicated
  `forms.py`.  Always set `success_url` or `get_success_url()`.
- Avoid `redirect('success_page')` тАУ this URL does not exist.  Redirect to
  the detail view of the saved object or back to the list.

`people` and `species` still use **function-based views** (FBVs).  Keep
the same redirect conventions.

### URL naming

| App       | `app_name` | Example name   |
|-----------|-----------|----------------|
| armor     | armor     | `armor:index`  |
| weapons   | weapons   | `weapons:index`|
| events    | events    | `events:index` |
| factions  | factions  | `factions:index`|
| worlds    | worlds    | `worlds:index` |
| ships     | ships     | `ships:index`  |
| vehicles  | vehicles  | `vehicles:index`|
| party     | *(none)*  | `party_index`  |
| people    | *(none)*  | `index`        |
| species   | *(none)*  | `index`        |
| mindmaps  | *(none)*  | `mindmap_index`|

Use `{% url 'app_name:view_name' %}` in templates for namespaced apps.

### Models and Traits

`Person.traits` is a **ManyToManyField** to `Trait`.  The old boolean columns
(`agile`, `cybernetic`, `tactician`, etc.) were removed in migration
`people/migrations/0015_тАж`.  Always use `person.traits.all()` for trait
checks, and use `prefetch_related('traits')` in list/detail views to avoid
N+1 queries.

### Forms

- `crispy_forms` with `crispy_bootstrap4` pack тАУ always use
  `{% load crispy_forms_tags %}` and `{{ form|crispy }}` or
  `{{ form.field|as_crispy_field }}`.
- Do not render form fields that no longer exist on the model.

## 4. Placeholder Apps

`ships` and `vehicles` have templates and URL confs but **no database
models or migrations**.  Their views return empty object lists.  When
implementing them, add models, migrations, and replace the placeholder
views with standard CBVs.

## 5. Google Sheets Integration

The canonical utility is **`cataclysm/cataclysm/utils/google_sheets.py`**.
Always import from `cataclysm.utils.google_sheets`.  The management command
`update_database_from_sheet` uses this to import crew from a Google Sheet.

### Environment variables

| Variable | Required | Description |
|---|---|---|
| `SERVICE_ACCOUNT_JSON` | one of the two | Full service-account key as a JSON string. Checked first; useful in containers where mounting a file is inconvenient. |
| `SERVICE_ACCOUNT_FILE` | one of the two | Path to the service-account JSON key file (default: `<BASE_DIR>/cataclysm/secrets/service_account.json`). Used when `SERVICE_ACCOUNT_JSON` is not set. |
| `SPREADSHEET_ID` | no | Google Sheets spreadsheet ID to import from (default: the hardcoded crew sheet ID). |
| `SHEET_RANGE` | no | Cell range to read, e.g. `Other Crew!A5:Z` (default: `Other Crew!A5:Z`). |

The secrets directory and JSON files are git-ignored.

## 6. Running Tests

```bash
cd cataclysm
python manage.py test
```

Tests live in each app's `tests.py` and cover basic view smoke tests
(list, add, detail return HTTP 200, and 404 on missing objects).
